# UML统一建模语言介绍

## 1. UML概述

**UML (Unified Modeling Language)** - 统一建模语言，是一种用于可视化、规格化、构建和文档化软件系统的人工制品的标准化语言。

### 1.1 UML发展历史

| 年份 | 事件 |
|------|------|
| 1994 | Grady Booch和James Rumbaugh在Rational Software开始统一方法 |
| 1995 | Ivar Jacobson加入，统一为UML |
| 1997 | UML 1.0被OMG采纳为标准 |
| 2005 | UML 2.0发布 |
| 2017 | UML 2.5发布 |

### 1.2 UML分类

```
UML
├── 结构型建模 (Structural Modeling)
│   ├── 类图 (Class Diagram)
│   ├── 对象图 (Object Diagram)
│   ├── 包图 (Package Diagram)
│   ├── 组件图 (Component Diagram)
│   ├── 部署图 (Deployment Diagram)
│   └── 组合结构图 (Composite Structure Diagram)
│
└── 行为型建模 (Behavioral Modeling)
    ├── 用例图 (Use Case Diagram)
    ├── 活动图 (Activity Diagram)
    ├── 状态机图 (State Machine Diagram)
    ├── 顺序图 (Sequence Diagram)
    ├── 通信图 (Communication Diagram)
    ├── 时序图 (Timing Diagram)
    └── 交互概观图 (Interaction Overview Diagram)
```

## 2. 类图 (Class Diagram)

类图是面向对象建模的核心，用于展示类的结构以及类之间的关系。

### 2.1 类的表示

```
┌─────────────────────────────────┐
│         ClassName               │  ← 类名（粗体居中）
├─────────────────────────────────┤
│ - field1: Type                  │  ← 属性
│ - field2: Type                  │
├─────────────────────────────────┤
│ + method1(): ReturnType         │  ← 方法
│ + method2(param: Type): ReturnType
└─────────────────────────────────┘
```

### 2.2 访问修饰符

| 符号 | 含义 | 说明 |
|------|------|------|
| `+` | public | 公共 |
| `-` | private | 私有 |
| `#` | protected | 受保护 |
| `~` | package | 包级私有 |

### 2.3 类图示例

```
┌─────────────────────────┐
│         Person          │
├─────────────────────────┤
│ - name: String          │
│ - age: int              │
│ - address: Address      │
├─────────────────────────┤
│ + getName(): String     │
│ + setName(name: String)  │
│ + getAge(): int         │
│ + birthday(): void      │
└─────────────────────────┘
```

## 3. 类之间的关系

### 3.1 继承/泛化 (Inheritance)

**概念**：子类继承父类的属性和方法

```
        ┌───────────┐
        │   父类    │
        │  Parent   │
        └─────┬─────┘
              │
              │ 空心三角箭头 + 实线
              ▼
        ┌───────────┐
        │   子类    │
        │  Child    │
        └───────────┘
```

**代码示例**：
```java
public class Parent {
    public String name;
    public void display() { }
}

public class Child extends Parent {
    // 自动继承name属性和display()方法
}
```

### 3.2 实现 (Realization)

**概念**：类实现接口

```
        ┌───────────┐
        │  <<interface>>  │
        │   接口     │
        └─────┬─────┘
              │
              │ 空心三角箭头 + 虚线
              ▼
        ┌───────────┐
        │   实现类   │
        └───────────┘
```

**代码示例**：
```java
public interface Drawable {
    void draw();
}

public class Circle implements Drawable {
    @Override
    public void draw() {
        // 实现draw方法
    }
}
```

### 3.3 关联 (Association)

**概念**：类之间有某种联系，是双向的

```
        ┌───────────┐          ┌───────────┐
        │   教师    │──────────│   学生    │
        │  Teacher  │  1    *  │  Student  │
        └───────────┘          └───────────┘
```

**代码示例**：
```java
public class Teacher {
    private Student[] students;  // 一个教师对应多个学生
}

public class Student {
    private Teacher teacher;  // 一个学生对应一个教师
}
```

### 3.4 聚合 (Aggregation)

**概念**：整体与部分的关系，但部分可以独立于整体存在

```
        ┌───────────┐
        │   汽车    │◇─────────│   车轮    │
        │   Car     │  整体○─部分 │  Wheel   │
        └───────────┘          └───────────┘
```

**代码示例**：
```java
public class Car {
    private Wheel[] wheels;  // 汽车有车轮，但车轮可以独立存在
}
```

### 3.5 组合 (Composition)

**概念**：整体与部分的关系，但部分不能独立于整体存在

```
        ┌───────────┐
        │   汽车    │◆─────────│   引擎    │
        │   Car     │  整体●─部分 │  Engine  │
        └───────────┘          └───────────┘
```

**代码示例**：
```java
public class Car {
    private Engine engine;  // 汽车有引擎，引擎不能独立于汽车存在

    public Car() {
        this.engine = new Engine();  // 引擎随汽车创建而创建
    }
}
```

### 3.6 依赖 (Dependency)

**概念**：一个类的实现依赖于另一个类

```
        ┌───────────┐          ┌───────────┐
        │   订单    │──────────│   日期    │
        │  Order   │  依赖      │   Date    │
        └───────────┘          └───────────┘
```

**代码示例**：
```java
public class Order {
    // 订单的某个方法使用了Date作为参数或局部变量
    public boolean isDeliveryValid(Date deliveryDate) {
        return deliveryDate.after(new Date());
    }
}
```

### 3.7 关系对比

| 关系类型 | 符号 | 特点 | 代码表现 |
|---------|------|------|---------|
| 继承 | ──▷ | is-a | extends |
| 实现 | ──▷ | can-do | implements |
| 关联 | ── | has-a | 成员变量 |
| 聚合 | ◇─ | has-a (可分离) | 成员变量 |
| 组合 | ◆─ | has-a (不可分) | 成员变量+创建 |
| 依赖 | ──▶ | uses-a | 方法参数/局部变量 |

## 4. 多重性 (Multiplicity)

表示一个类实例可以关联到另一个类实例的数量。

| 表示法 | 含义 |
|--------|------|
| `1` | 有且只有一个 |
| `0..1` | 零个或一个 |
| `*` | 零个或多个 |
| `1..*` | 一个或多个 |
| `2..5` | 指定范围 |

**示例**：
```
┌───────────┐         ┌───────────┐
│   课程    │ 1    *  │   学生    │
│  Course   │─────────│  Student  │
└───────────┘         └───────────┘
一门课程对应多个学生，一个学生也可以上多门课程
```

## 5. 抽象类和接口

### 5.1 抽象类

```
┌─────────────────────────────────┐
│      <<abstract>>               │
│         Animal                  │
├─────────────────────────────────┤
│ # name: String                  │
├─────────────────────────────────┤
│ + abstract void speak()         │
│ + void breathe()               │
└─────────────────────────────────┘
```

**代码示例**：
```java
public abstract class Animal {
    protected String name;

    public abstract void speak();  // 抽象方法，子类必须实现

    public void breathe() {       // 具体方法，子类可直接使用
        System.out.println(name + "正在呼吸");
    }
}
```

### 5.2 接口

```
┌─────────────────────────────────┐
│      <<interface>>              │
│         Flyable                 │
├─────────────────────────────────┤
│ + fly(): void                   │
└─────────────────────────────────┘
```

**代码示例**：
```java
public interface Flyable {
    void fly();
}

// 实现接口
public class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("鸟儿飞翔");
    }
}
```

## 6. 设计原则对应的UML表示

### 6.1 单一职责原则 (SRP)

一个类只有一个引起变化的原因。

```
                    ┌─────────────────┐
                    │     Button       │
                    │   (按钮类)       │
                    ├─────────────────┤
                    │ + onClick()      │
                    │ + render()       │
                    └─────────────────┘
                            │
           ┌────────────────┼────────────────┐
           │                │                │
           ▼                ▼                ▼
    ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
    │  ClickHandler│  │  Renderer   │  │  StateManager│
    │ (点击处理)   │  │  (渲染器)    │  │  (状态管理)  │
    └─────────────┘  └─────────────┘  └─────────────┘
```

### 6.2 开闭原则 (OCP)

对扩展开放，对修改关闭。

```
                        ┌─────────────────┐
                        │   Shape          │
                        │  <<abstract>>   │
                        ├─────────────────┤
                        │ + draw()         │
                        └────────┬────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
     ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
     │  Circle     │    │  Rectangle  │    │   Triangle  │
     └─────────────┘    └─────────────┘    └─────────────┘
```

### 6.3 依赖倒置原则 (DIP)

高层模块不应该依赖低层模块，两者都应该依赖抽象。

```
         高层模块                         低层模块
     ┌─────────────────┐             ┌─────────────────┐
     │   Notification   │             │    Email         │
     │  (通知服务)       │────────────▶│   (邮件)         │
     └────────┬────────┘             └─────────────────┘
              │                                ▲
              │                                │
              │        ┌─────────────────┐      │
              └───────▶│  <<interface>>  │──────┘
                       │   MessageSender │
                       ├─────────────────┤
                       │ + send()        │
                       └─────────────────┘
```

## 7. 设计模式对应的UML

### 7.1 单例模式 (Singleton)

```
┌─────────────────────────────────┐
│        <<unique>>               │
│        Singleton                │
├─────────────────────────────────┤
│ - instance: Singleton           │
├─────────────────────────────────┤
│ - Singleton()                   │
│ + getInstance(): Singleton      │
│ + operation()                   │
└─────────────────────────────────┘
```

### 7.2 工厂方法模式

```
                    ┌─────────────────┐
                    │    Blacksmith    │
                    │  (抽象创建者)     │
                    ├─────────────────┤
                    │ + createWeapon() │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
     ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
     │ElfBlacksmith│ │OrcBlacksmith│ │  ...        │
     └─────────────┘ └─────────────┘ └─────────────┘
```

### 7.3 策略模式

```
┌─────────────────────────────────┐
│         Context                 │
├─────────────────────────────────┤
│ - strategy: Strategy            │
├─────────────────────────────────┤
│ + setStrategy(s: Strategy)      │
│ + execute()                    │
└───────────────┬─────────────────┘
                │
                │ uses
                ▼
    ┌─────────────────────────────────┐
    │      <<interface>>              │
    │        Strategy                │
    ├─────────────────────────────────┤
    │ + execute()                    │
    └─────────────┬───────────────────┘
                  │
    ┌─────────────┼─────────────┐
    │             │             │
    ▼             ▼             ▼
┌─────────┐ ┌─────────┐ ┌─────────────┐
│StrategyA│ │StrategyB│ │  StrategyN  │
└─────────┘ └─────────┘ └─────────────┘
```

### 7.4 装饰器模式

```
┌─────────────────────────────────┐
│      <<interface>>              │
│        Component                │
├─────────────────────────────────┤
│ + operation()                   │
└─────────────┬───────────────────┘
              │
      ┌───────┴───────┐
      │               │
      ▼               ▼
┌───────────┐   ┌───────────────────┐
│ Concrete  │   │    Decorator       │
│Component  │   │   <<abstract>>     │
└───────────┘   ├───────────────────┤
                │ - component       │
                ├───────────────────┤
                │ + operation()     │
                └────────┬──────────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌───────────┐  ┌───────────┐  ┌───────────┐
    │DecoratorA │  │DecoratorB │  │DecoratorC │
    └───────────┘  └───────────┘  └───────────┘
```

## 8. 时序图 (Sequence Diagram)

时序图展示对象之间的交互顺序。

### 8.1 基本元素

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ Client  │     │  Order  │     │ Payment │
└────┬────┘     └────┬────┘     └────┬────┘
     │              │              │
     │ 1. create()  │              │
     │─────────────▶│              │
     │              │              │
     │ 2. process() │              │
     │─────────────▶│              │
     │              │              │
     │              │ 3. pay()     │
     │              │─────────────▶│
     │              │              │
     │              │ 4. result     │
     │              │◀─────────────│
     │              │              │
     │ 5. confirm   │              │
     │◀─────────────│              │
     │              │              │
```

### 8.2 生命线和激活条

```
    对象              生命线           对象
     │                 │               │
     │    ┌─────┐      │               │
     │    │     │      │               │
     │    │     │      │               │
     │    └─────┘      │               │
     │      │          │               │
     │      ▼          │               │
     │   激活条         │               │
     │      │          │               │
     │      │◀─────────│───返回────────│
     │      │          │               │
     │      ▼          │               │
     │   激活条结束     │               │
     │                 │               │
```

## 10. 总结

理解UML对于学习和应用设计模式至关重要：

1. **类图**是最常用的，用于展示类结构和类之间的关系
2. **时序图**用于展示对象交互顺序
3. **掌握关系符号**：继承、实现、关联、聚合、组合、依赖
4. **多重性**帮助理解对象数量关系
5. **结合代码**理解UML，而不是孤立的记忆符号
